ジメジメする季節になってきたので初投稿です。

MOSH では、GitHub Actions の security hardening の一環として [zizmor](https://github.com/zizmorcore/zizmor) を導入しました[^1]。

https://docs.zizmor.sh/

この記事では、なぜ zizmor を入れたのか、実際に導入しての考慮ポイントなどを整理します。

## なぜzizmorを入れるのか

MOSH では CI/CD として GitHub Actions を利用しています。Actions のお困りポイントとして、workflow の書き方次第で攻撃面が広がりやすいというものがあります。

デフォルトのままだとかなり permissive なのが問題で、実際最近のサプライチェーン攻撃の入り口としてよく使われてもいます。

弊社ではすでに pinact を使って third-party action の参照を commit SHA に固定していました。これは supply chain attack 対策として有効ですが、workflow 自体の設定ミスまでは見てくれません。

例えば以下のようなものです。

- `GITHUB_TOKEN` や GitHub App token の過剰な権限
- `actions/checkout` 後に credential が残り続ける設定
- `pull_request_target` や `workflow_run` など、強い権限で走り得る trigger の使い方
- 未信頼入力の script 内展開
- etc...

そこで導入したのが zizmor です。

zizmor は GitHub Actions workflow 向けの静的解析ツールです。workflow YAML や composite action を解析し、上に挙げたような workflow 固有のリスクを検知できます。

また、GitHub annotation や SARIF upload に対応しており、CI 上で結果を受け取りやすいです。`zizmor: ignore[...]` による例外指定もでき、`persona` や `min-severity` で検知の厳しさも調整できます。細かいところの体験がうまく作られていて、小町的にポイント高いです。

最近自分がメンテしている OSS でも zizmor を導入する機会があり、これを社内にも広げようと思い今回導入を決意しました[^2]。

## 選定時に考えたこと

導入する形として、pinact と zizmor をまとめた reusable workflow を用意し、各 repository から呼び出す方針にしています。ここではそれを含め、導入時の考慮ポイントを整理します。

### zizmorだけではcommit SHA修正まで自動化できない

zizmor 導入以前は pinact で commit SHA pinning を担保していました。そして zizmor も未固定の `uses:` を検知できます。

ただし、検知した未固定 action を commit SHA へ自動置換する運用は pinact に任せる方が簡単です。zizmor は workflow の危険な書き方を検知する役割、pinact は SHA pinning の修正を担う役割、と分けた方が運用しやすいのでそのようにしました。

### 検知した結果をどう受け取るか

zizmor は SARIF upload (GitHub Advanced Security) による運用もできますが、今回は GitHub annotation とログで結果を確認する運用を採用しました。

理由として、private repository で GitHub Advanced Security を前提にすると、課金や repository ごとの運用差分が増えてしまう、というのがありました。

annotation は、出力数に上限があるなど一部制限があります。ただ今回は目的を達成するうえで SARIF upload と annotation の差が大きくないと判断しました。そのため、無料で使える annotation を選定しています。

これは zizmor-action の input で設定できます: https://github.com/zizmorcore/zizmor-action#annotations

## 実際に直したこと

ここでは実際に zizmor のおかげで修正できた部分をいくつかピックアップして紹介します。

### `persist-credentials: false`を入れる

かなり多く出たのが `actions/checkout` の `persist-credentials: false` です。

```yaml
- uses: actions/checkout@...
  with:
    persist-credentials: false
```

`actions/checkout` はデフォルトだと credential を git config に残します。後続 step で push が必要ないなら残す必要はありません。push が必要な workflow では、用途に応じた GitHub App token を明示的に渡すようにしました。

この問題の詳細については、最近 koki さんが書かれた記事も話題になっていました。詳しくはこちらも読むと面白いです:

https://zenn.dev/kou_pg_0131/articles/gha-checkout-persist-credentials

### permissionsをjob単位に寄せる

元々 workflow top-level に広めの `permissions` が置かれている箇所がありました。

```yaml
permissions:
  id-token: write
  contents: read
  pull-requests: write
```

これだと、すべての job が同じ権限を持つことになってしまい最小権限の原則を満たせていません。そこで可能な箇所では top-level を `permissions: {}` や `contents: read` 程度に寄せ、実際に必要な job にだけ権限を付けるようにしました。

```yaml
permissions: {}

jobs:
  job1:
    permissions:
      id-token: write # ◯◯するのに必要
      contents: read
```

コメントも少し入れています。権限コメントはなくても動きますが、`contents: write` や `pull-requests: write` のような強めの権限は、後から見たとき必要な理由を確認できる方がレビューしやすいので入れています。

### template expressionをshellに直接埋め込まない

`github.event.*` などの値は、PR title や branch name のように user-controlled な値を含むことがあります。それを shell script に直接埋め込むと、script injection の余地が生まれてしまいます。

例えばこういうものです。

```yaml
run: |
  if [[ "${{ github.event.release.tag_name }}" =~ ^[0-9]{12}$ ]]; then
    echo "yes"
  fi
```

このような箇所は env 経由に寄せました。

```yaml
env:
  RELEASE_TAG_NAME: ${{ github.event.release.tag_name }}
run: |
  if [[ "$RELEASE_TAG_NAME" =~ ^[0-9]{12}$ ]]; then
    echo "yes"
  fi
```

### ignoreには理由を残す

ほとんどの指摘は妥当な一方で、false positive だったり何らかの理由があったりで即座には対応できない・しない箇所もありました。

そういうところでは `zizmor: ignore[...]` を使い ignore できます。これに付随して理由も記載するようにしました。

```yaml
on:
  workflow_run: # zizmor: ignore[dangerous-triggers] ◯◯するのに必要
```

ignore は便利ですが、後から見た人が歴史をさっと理解できないと負債になり得ます。ここはソースコード内と同じで、文脈を残しておくのが大事そうです。

## これから試したい部分

今回導入した zizmor は `persona` や `min-severity` などで厳密性を調整できるのですが、今回は一旦デフォルトのまま導入しています。今後運用が安定したタイミングでより厳密にするなど MOSH に最適な設定値を探っていきたいなあと考えています。

また、似たような静的解析ツールとして actionlint や ghalint などが存在していますが、今回は zizmor のみ導入する形としました。理由として指摘箇所が膨大になることが予想されそれぞれのツールの特性を評価しづらくなること、運用負荷の予想がしづらいこと、などがありました。とはいえそれぞれのツールにしかないルールや利点なども存在するので今後の併用も検討しています。
これらをいい感じに scaffold した ghasec というツールもあるのでこちらも気になります:

https://zenn.dev/kou_pg_0131/articles/ghasec-introduction

## まとめ

社内 repository に zizmor を導入し、既存の pinact と合わせて運用を開始しました。

冒頭で触れた supply chain attack は、主に public repository に関する話が多いです。

一方で、AI による attack vector の変化や Actions の利便性ゆえの責務の拡大などを鑑みると、security hardening は必須になってきていると感じています。

GitHub 公式の機能としても commit SHA pinning enforcement などが導入されてきている一方、依然として third-party tools に頼る部分も大きいです。この構造にはある種の不安定さも存在するので GitHub の機能だけでカバーできる部分も増えていってほしいなあと思います。
